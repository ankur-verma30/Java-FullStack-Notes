Swagger is an open-source framework used for designing, building, documenting, and consuming RESTful APIs.

It provides tools to:
‚û§ Describe your API in a standardized format (called OpenAPI Specification, formerly called Swagger Specification).
‚û§ Generate interactive API documentation automatically.
‚û§ Test APIs easily from a browser.
‚û§ Auto-generate client SDKs and server stubs in multiple languages.

3Ô∏è‚É£ Example REST controller
package com.demo.controller;

import org.springframework.web.bind.annotation.*;
import java.util.*;
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.Parameter;
import io.swagger.v3.oas.annotations.media.Content;
import io.swagger.v3.oas.annotations.media.Schema;
import io.swagger.v3.oas.annotations.responses.ApiResponse;
import io.swagger.v3.oas.annotations.responses.ApiResponses;

@RestController
@RequestMapping("/api/students")
public class StudentController {

    private final Map<Integer, String> studentDB = new HashMap<>();

    // ‚úÖ GET all students
    @Operation(summary = "Get all students", description = "Fetches the list of all students in the database.")
    @ApiResponse(responseCode = "200", description = "List retrieved successfully")
    @GetMapping
    public Collection<String> getAllStudents() {
        if (studentDB.isEmpty()) {
            studentDB.put(1, "Ankur Verma");
            studentDB.put(2, "John Doe");
        }
        return studentDB.values();
    }

    // ‚úÖ GET student by ID
    @Operation(summary = "Get a student by ID", description = "Fetch a specific student using their ID.")
    @ApiResponses({
        @ApiResponse(responseCode = "200", description = "Student found",
            content = @Content(mediaType = "application/json",
            schema = @Schema(implementation = String.class))),
        @ApiResponse(responseCode = "404", description = "Student not found")
    })
    @GetMapping("/{id}")
    public String getStudentById(
        @Parameter(description = "ID of the student to fetch", example = "1")
        @PathVariable int id) {

        return studentDB.getOrDefault(id, "Student not found");
    }

    // ‚úÖ POST add a new student
    @Operation(summary = "Add a new student", description = "Create a new student record in the database.")
    @ApiResponse(responseCode = "201", description = "Student added successfully")
    @PostMapping
    public String addStudent(@RequestParam int id, @RequestParam String name) {
        studentDB.put(id, name);
        return "Added student: " + name;
    }

    // ‚úÖ DELETE student by ID
    @Operation(summary = "Delete a student", description = "Remove a student by their ID.")
    @ApiResponse(responseCode = "200", description = "Student deleted successfully")
    @DeleteMapping("/{id}")
    public String deleteStudent(@PathVariable int id) {
        if (studentDB.remove(id) != null) {
            return "Deleted student with ID: " + id;
        }
        return "Student not found!";
    }
}

üîÑ Annotations used in Swagger
üß© 1Ô∏è‚É£ CLASS LEVEL ANNOTATIONS
============================================================
Annotation       | Purpose
------------------------------------------------------------
üîó @Tag             | Defines a logical grouping of APIs under a
                 | name and description. Used at class level to
                 | describe a controller or API section.
------------------------------------------------------------
Example:
@RestController
@RequestMapping("/api/students")
@Tag(name = "Student API",
     description = "Operations related to students")
public class StudentController { }

------------------------------------------------------------
üîó @OpenAPIDefinition | Defines global metadata for the entire API
                   | ‚Äî title, version, description, contact,
                   | license, etc. Usually placed on main
                   | Application or Config class.
------------------------------------------------------------
Example:
@OpenAPIDefinition(
    info = @Info(
        title = "Student Management API",
        version = "1.0",
        description = "API documentation for Student Management"
    )
)
@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args){
        SpringApplication.run(DemoApplication.class, args);
    }
}

------------------------------------------------------------
üîó @Server          | Defines server URL and description for the
                 | API. Multiple can be declared for different
                 | environments.
------------------------------------------------------------
Example:
@OpenAPIDefinition(
    servers = {
        @Server(
            url = "http://localhost:8080",
            description = "Local Server"
        )
    }
)
============================================================


üß© 2Ô∏è‚É£ METHOD LEVEL ANNOTATIONS
============================================================
Annotation       | Purpose
------------------------------------------------------------
üîó @Operation       | Describes an individual API operation
                 | (summary, description, parameters, responses).
------------------------------------------------------------
Example:
@PostMapping("/add")
@Operation(
    summary = "Add a new student",
    description = "Creates a new student record using details"
)
public String addStudent(@RequestParam int id,
                         @RequestParam String name) {
    return "Added student " + name;
}

------------------------------------------------------------
üîó @ApiResponse     | Describes possible HTTP responses for an
                 | operation (e.g., 200 OK, 404 Not Found).
------------------------------------------------------------
Example:
@ApiResponses({
    @ApiResponse(responseCode = "200",
                 description = "Student added successfully"),
    @ApiResponse(responseCode = "400",
                 description = "Invalid input")
})

------------------------------------------------------------
üîó @RequestBody     | Marks a parameter as coming from the request
                 | body (for JSON input).
------------------------------------------------------------
Example:
@PostMapping("/register")
public String registerStudent(@RequestBody Student student) {
    return "Registered " + student.getName();
}

------------------------------------------------------------
üîó @Parameter       | Describes a single parameter for documentation
                 | (overrides auto-generated info).
------------------------------------------------------------
Example:
@PostMapping("/greet")
public String greet(
    @Parameter(description = "Name of student to greet",
               example = "Ankur")
    @RequestParam String name) {
    return "Hello " + name;
}
============================================================


üß© 3Ô∏è‚É£ PARAMETER LEVEL ANNOTATIONS
============================================================
Annotation       | Purpose
------------------------------------------------------------
üîó @Parameter       | Provides documentation for individual params
                 | (path, query, header, etc.).
------------------------------------------------------------
Example:
@GetMapping("/find")
public String findStudent(
    @Parameter(description = "Student ID", example = "101")
    @RequestParam int id) {
    return "Found student " + id;
}

------------------------------------------------------------
üîó @Schema          | Defines schema metadata for models or params
                 | (example value, type, format, etc.).
------------------------------------------------------------
Example:
public class Student {
    @Schema(description = "Unique ID of the student",
            example = "1")
    private int id;

    @Schema(description = "Full name of the student",
            example = "Ankur Verma")
    private String name;
}
============================================================


üß© 4Ô∏è‚É£ MODEL / ENTITY LEVEL ANNOTATIONS
============================================================
Annotation       | Purpose
------------------------------------------------------------
üîó @Schema          | Provides details about model class or its
                 | fields. Appears in Swagger model schema.
------------------------------------------------------------
Example:
@Schema(name = "Student",
        description = "Represents a student in the system")
public class Student {
    @Schema(description = "Unique ID", example = "1")
    private int id;

    @Schema(description = "Name", example = "Ankur Verma")
    private String name;
}
============================================================


üß© 5Ô∏è‚É£ RESPONSE AND ERROR HANDLING ANNOTATIONS
============================================================
Annotation       | Purpose
------------------------------------------------------------
üîó @ApiResponse     | Describes a specific response for a given
                 | status code.
------------------------------------------------------------
Example:
@ApiResponse(responseCode = "404",
             description = "Student not found")

------------------------------------------------------------
üîó @ApiResponses    | Container for multiple @ApiResponse
                 | annotations.
------------------------------------------------------------
Example:
@ApiResponses({
    @ApiResponse(responseCode = "200",
                 description = "Success"),
    @ApiResponse(responseCode = "400",
                 description = "Bad Request")
})

4Ô∏è‚É£ Add Basic Metadata (Optional)
You can customize the Swagger page (title, description, version, contact info, etc.)
by creating a @Configuration class like this:
package com.example.demo.config;

import io.swagger.v3.oas.models.info.Info;
import io.swagger.v3.oas.models.info.Contact;
import io.swagger.v3.oas.models.OpenAPI;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class SwaggerConfig {

    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
            .info(new Info()
                .title("Student Management API")
                .version("1.0")
                .description("API documentation for Student Management System")
                .contact(new Contact()
                    .name("Ankur Verma")
                    .email("ankur@example.com")
                    .url("https://github.com/ankur-verma30")));
    }
}

5Ô∏è‚É£ Adding JWT or Basic Auth Support (Optional)
If your APIs require authentication, you can define security schemes in Swagger:
import io.swagger.v3.oas.models.security.SecurityScheme;
import io.swagger.v3.oas.models.security.SecurityRequirement;

@Bean
public OpenAPI secureOpenAPI() {
    return new OpenAPI()
        .addSecurityItem(new SecurityRequirement().addList("bearerAuth"))
        .components(new io.swagger.v3.oas.models.Components()
            .addSecuritySchemes("bearerAuth",
                new SecurityScheme()
                    .type(SecurityScheme.Type.HTTP)
                    .scheme("bearer")
                    .bearerFormat("JWT")))
        .info(new Info().title("Secure API").version("1.0"));
}
Now, Swagger UI will show an "Authorize" button to enter your JWT token.

üî∑ The OpenAPI Specification (OAS)
At the heart of Swagger lies the OpenAPI Specification ‚Äî a language-agnostic standard to describe REST APIs.

Example (in YAML or JSON):
openapi: 3.0.0
info:
  title: Student API
  version: 1.0.0
paths:
  /students:
    get:
      summary: Get all students
      responses:
        '200':
          description: Successful response

This specification defines:
‚úîÔ∏è Endpoints (/students)
‚úîÔ∏è HTTP methods (GET, POST, etc.)
‚úîÔ∏è Parameters and request bodies
‚úîÔ∏è Responses and status codes
‚úîÔ∏è Authentication methods
‚úîÔ∏è Data models (schemas)

üîÑ Swagger Ecosystem (Tools)
Swagger offers a suite of tools for API lifecycle management:
| Tool                           | Purpose                                                           |
| ------------------------------ | ----------------------------------------------------------------- |
| **Swagger Editor**             | Write and visualize the OpenAPI spec in the browser.              |
| **Swagger UI**                 | Generate interactive API documentation (try out APIs in browser). |
| **Swagger Codegen**            | Auto-generate client SDKs or server stubs from spec.              |
| **SwaggerHub**                 | Cloud-based platform to collaborate on API design & docs.         |
| **Swagger Parser / Validator** | Validate and parse OpenAPI documents.                             |


üîÑ Why Swagger is Used (Main Benefits)
1Ô∏è‚É£ Standardization
‚û§ Defines APIs in a consistent, machine-readable format using OpenAPI.
‚û§ Acts as a single source of truth for developers and consumers.

2Ô∏è‚É£ Documentation
‚û§ Automatically generates beautiful, interactive documentation.
‚û§ Users can try endpoints directly from the Swagger UI.

3Ô∏è‚É£ Communication & Collaboration
‚û§ Helps backend and frontend teams communicate API contracts early.
‚û§ Reduces miscommunication between teams.

4Ô∏è‚É£ Testing and Mocking
‚û§ You can test API endpoints directly from Swagger UI.
‚û§ Can create mock servers before backend implementation (useful for frontend devs).

5Ô∏è‚É£ Code Generation
Auto-generates client SDKs (Java, Python, JS, etc.) and server stubs, saving development time.

6Ô∏è‚É£ Version Control
Swagger specs are plain JSON/YAML, easy to version control via Git.

Use Cases of Swagger
| Use Case                        | Description                                                                |
| ------------------------------- | -------------------------------------------------------------------------- |
| **API Design First Approach**   | Design the API in Swagger before writing backend code.                     |
| **API Documentation**           | Generate interactive, shareable API docs automatically.                    |
| **Client SDK Generation**       | Generate code for calling APIs in various programming languages.           |
| **Testing APIs**                | Manually test API endpoints (GET, POST, PUT, DELETE) from the UI.          |
| **Mocking APIs**                | Simulate API behavior before the backend is ready.                         |
| **Contract-Driven Development** | Both frontend and backend developers agree on the Swagger spec (contract). |
| **Versioning and Maintenance**  | Easily manage changes in API versions and update docs automatically.       |
