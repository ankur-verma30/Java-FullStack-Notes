ğŸ”„ Kafka Topic:
â¤ Named container for similar events, unique identifier of a topic is its name.
âœ… Example: Student topic will have student related data.
â¤ They are like tables in a database.
â¤ They live insde a broker.
â¤ Producer produce a message into the topic(ultimately to partitions in round robin fashion) or direectly to the partitions. Consumer poll continuously for few message using the topic name.

âœ”ï¸ Partition:A topic is partitioned and distributed to kafka brokers in round robin fashion to achieve distributed system.
âœ”ï¸ Replication Factor: A partition is replicated by this factor and it is replicated in another broker to prevent fault tolerance.


ğŸ”„ Partitions
â¤ A topic is split into serveral parts which are known as the partitions of the topic.
â¤ Partitions is where actually the message is located inside the topic.
â¤ Therefore, while creating a topic we need to specify the number of partitions (the number is arbitrary and depends on the use case).
â¤ Each partition is an ordered, immutable sequence of messages that is continually appended toâ€”a structured commit log.
â¤ Each message gets stored into partitions with an incremental id known as the offset valu.
â¤ Ordering is there only at partition level(so if data is to be stored in order than do it on same partition).
â¤ Partition continuously grows(offset increases) as new records are produced.
â¤ All the records exist in distributed log file.

ğŸ”„ Working of Partitioner
We can send message with key or without key.
â¤ If we send message without key, the producer will send the message to the partitions in round robin fashion hence, we cannot gurantee the ordering of message as consumer poll the message from all the partitions at the same time.
â¤ If we send message with key, the producer will hash the key and then decide the partition to which the message is to be sent and the ordering will be maintained.
