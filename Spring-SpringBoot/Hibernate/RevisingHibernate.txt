üî∑ Timestamping in Data JPA
These are the two annotations used to automatically add creation and update timestamps to your entities:

üîó @CreationTimestamp
‚û§ Automatically sets the current timestamp when the entity is first persisted (inserted).
‚û§ It‚Äôs set only once, when the record is created.
‚û§ Commonly used for fields like createdAt, createdDate, or registeredOn

if we want to prevent the updation of the createdAt field, we can use @Column(updatable = false)

üëâ  When you save a new User, Hibernate automatically fills createdAt with the current time.

üîó @UpdateTimestamp
‚û§ Automatically updates the timestamp whenever the entity is updated (i.e., on UPDATE operation).
‚û§ Commonly used for fields like updatedAt, lastModified, or lastUpdatedOn.

if we want to prevent the insertion of the updatedAt field at the time of creation of record, we can use @Column(insertable = false)

üëâ When you modify any field in the User and save/update it, Hibernate automatically updates updatedAt to the current time.

üîÑ Behind the Scenes
‚û§ These are Hibernate-only annotations (not part of pure JPA).
‚û§ Hibernate injects the timestamp values before persisting or updating.
‚û§ No need to manually set createdAt or updatedAt in code.

üîÑ OneToOne Mapping
1Ô∏è‚É£ One Patient has exactly one Insurance, and one Insurance belongs to exactly one Patient.

2Ô∏è‚É£ Entity Roles
| Role             | Entity          | Description                                                                      |
| ---------------- | -----------     | -------------------------------------------------------------------------------- |
| Owning Side      | `Patient`       | The side that owns the foreign key (`patient_insurance_id`) in the database.     |
| Inverse Side     | `Insurance`     | The side that refers back to the owning side using `mappedBy`.                   |

3Ô∏è‚É£ Code Explanation
üîó In Patient (Owning side)
@OneToOne
@JoinColumn(name = "patient_insurance_id")
private Insurance insurance;


‚úÖ Key Points:
‚úîÔ∏è @OneToOne defines a 1‚Äì1 relationship.
‚úîÔ∏è @JoinColumn(name="patient_insurance_id") means the patient table will have a foreign key column named patient_insurance_id which points to the insurance table‚Äôs primary key (id).
‚úîÔ∏è This side controls the relationship ‚Äî insert/update/delete will happen through Patient.

üîó In Insurance (Inverse side)
@OneToOne(mappedBy = "insurance")
private Patient patient;

‚úÖ Key Points:
‚úîÔ∏è mappedBy = "insurance" ‚Üí tells JPA that Patient owns the relationship (the field name insurance in the Patient entity is the mapping point).
‚úîÔ∏è No separate foreign key column will be created in the insurance table.
‚úîÔ∏è It‚Äôs just a back-reference used for bidirectional navigation.

4Ô∏è‚É£ How It Works in Practice
‚û§ When Saving:
If you save a Patient with an associated Insurance,
JPA automatically updates the foreign key column (patient_insurance_id) in the patient table.

‚úÖ Example:
Insurance insurance = new Insurance();
insurance.setPolicyNumber("POL123");
insurance.setProvider("ICICI Health");

Patient patient = new Patient();
patient.setName("Isha Gupta");
patient.setEmail("isha@gmail.com");
patient.setInsurance(insurance);

patientRepository.save(patient); // Patient saved ‚Üí insurance linked

üëâ Cascade (optional):	You can use @OneToOne(cascade = CascadeType.ALL) if you want saving/deleting Patient to automatically affect Insurance.
üëâ Fetch Type:	Default is EAGER for @OneToOne. Can set to LAZY for better performance.

üîÑ ManyToOne Mapping(Appointment <-> Patient)
| Aspect                          | Explanation                                                                                                                         |
|Entities Involved                    | `Appointment` (many) ‚Üî `Patient(one)                                                                                            |
| Meaning                         | A single patient can have multiple appointments, but each appointment belongs to exactly one patient.                               |
| Owning Side                     | `Appointment` is the owning side because it contains the foreign key column `patient_id`.                                       |
|Foreign Key              | The `@JoinColumn(name="patient_id")` annotation creates a foreign key in the `appointment` table that references the `patient` table‚Äôs `id` |
| Nullable = false                | Ensures that every appointment must have an associated patient ‚Äî an appointment cannot exist without a patient.                 |
|Inverse Relationship(Optional)| Patient`class, define the reverse mapping using `@OneToMany(mappedBy = "patient")` if we want to access all appointments of a patient  |
| Database Table Structure        | The `appointment` table will have columns like `id`, `appointment_time`, `reason`, and `patient_id` (FK referencing `patient.id`).  |
| Example    | Patient ‚ÄúJohn‚Äù can have appointments like ‚ÄúGeneral Checkup‚Äù and ‚ÄúFollow-up Visit‚Äù. Both appointments will have the same `patient_id` (John‚Äôs ID).        |

üîÑ How to write custom Repository Methods for JPA 
Rules for writing custom methods:

‚ö° Start with a Recognized Prefix
‚û§ Every finder method must start with one of these prefixes:
üîó findBy... ‚Üí fetch data
üîó readBy... ‚Üí alias of findBy
üîó queryBy... ‚Üí alias of findBy
üîó getBy... ‚Üí alias of findBy
üîó countBy... ‚Üí count records matching condition
üîó existsBy... ‚Üí check if record exists
üîó deleteBy... / removeBy... ‚Üí delete records

‚ö° End with an Entity Name
‚û§ The entity name must be the last word in the method name.

‚ö° Method name should follow camelCase
‚úÖExample:
Optional<Employee> findByEmailId(String email);
long countByLastName(String lastName);
boolean existsByMobile(String mobile);

‚û§ Avoid using special characters in the method name.
‚û§ Avoid using reserved words in the method name.
‚û§ Avoid using SQL keywords in the method name.

‚úîÔ∏è If want to create method based on more then one fieldName then:
‚ö°  Rules to aggregate multiple field values:
‚û§ Using AND: findByEmailIdAndMobile(String email,String mobile);
‚û§ Using OR: findByEmailIdOrMobile(String email,String mobile);

‚úÖExample:
Optional<Employee> findByEmailIdAndMobile(String email,String mobile);

üåü To Read More on Naming and Method making visit: https://docs.spring.io/spring-data/jpa/reference/jpa/query-methods.html

üîÑ Native Query
A native query means you are writing the raw SQL that will be executed directly by the database, rather than using JPQL (Java Persistence Query Language).

@Query(value = "SELECT * FROM patient", nativeQuery = true)
List<Patient> findAllPatient();
üëâ This tells Spring Data JPA:
    ‚úîÔ∏è Execute the raw SQL:
        üîó SELECT * FROM patient;
‚û§ Map the result to the Patient entity.
‚û§ Because you marked nativeQuery = true, JPA will not validate the query against entity names or JPQL syntax.

üîÑ How It Differs from JPQL
| Feature     | JPQL                                 | Native Query                                 |
| ----------- | ------------------------------------ | -------------------------------------------- |
| Syntax      | Works on entity names and fields | Works on actual table and column names   |
| Example     | `SELECT p FROM Patient p`            | `SELECT * FROM patient`                      |
| Validation  | Checked by JPA at startup            | Not checked (errors show at runtime)         |
| Portability | Database-agnostic                    | Database-specific SQL                        |
| Mapping     | Automatic (based on entity fields)   | Must ensure column names match entity fields |


üîÑ When to Use Native Queries
‚úÖ Use native queries when:
    ‚úîÔ∏è You need database-specific functions (like LIMIT, TOP, or joins not supported by JPQL).
    ‚úîÔ∏è You‚Äôre dealing with complex queries or performance tuning.
    ‚úîÔ∏è You want to reuse an existing SQL query or stored procedure.

‚ùå Avoid them when:
    ‚úîÔ∏è A simple JPQL query or derived method (findByXyz) would work ‚Äî it‚Äôs safer and portable.

üß† Concept Behind CascadeType and FetchType

üîÑ Cascade Type ‚Äî ‚ÄúWhat happens to related entities when one entity is saved/updated/deleted‚Äù
‚û§ When you perform operations on one entity (like save, delete, update), cascade determines whether those operations should propagate to the associated entity.

‚úÖ Example from createInsurance() test:
patient.setInsurance(insurance);
insurance.setPatient(patient);
patientRepository.save(patient); //Optional will saved automatically when the transaction ends because of CascadeType.ALL

‚û§ Here, you are saving the patient, but the insurance is a related entity.
‚ùì Now the question is: Should the insurance automatically be saved when I save the patient?
‚û§ That‚Äôs where CascadeType comes in.


üîÑ Common Cascade Types
| Cascade Type          | What it does                                                         |                     Example Use Case                                  |
|CascadeType.PERSIST| When you save(`persist`)a parent entity, the child entity is also saved. | Save a `Patient` and automatically save its `Insurance`.              |
|CascadeType.MERGE | When you update (`merge`) a parent entity, updates propagate to the child.| Update `Patient` details and automatically update linked `Insurance`. |
|CascadeType.REMOVE   | Deleting parent deletes child.                                         | Delete a `Patient`, also delete `Insurance`.                          |
|CascadeType.ALL`     | Applies all above operations.                                          | When you want total propagation of all operations.                    |
|CascadeType.DETACH`  | Detaches both parent and child from the persistence context.           | Rarely used.                                                          |
|CascadeType.REFRESH` | Refreshing parent also refreshes the child.                            | Reload from DB both entities.                                         |


‚úÖ Correct Way to Use in Your Entities
üîÑ In your Patient entity:

@OneToOne(cascade = CascadeType.ALL)
@JoinColumn(name="patient_insurance_id")
private Insurance insurance;

‚û°Ô∏è This means:
‚û§ When you call patientRepository.save(patient), the insurance will automatically be persisted ‚Äî even if you don‚Äôt call insuranceRepository.save() manually.

‚û§ So in your test:
Insurance insurance = ...
Patient patient = insuranceService.assignInsuranceToPatient(insurance, 1L);

üëâ If you had cascade = CascadeType.ALL, you could skip manually saving insurance ‚Äî the cascade would handle it.

üîÑ Fetch Type ‚Äî ‚ÄúWhen is the related entity loaded from the database?‚Äù
Every JPA relationship has a fetch strategy that decides whether related entities are loaded immediately (EAGER) or lazily (LAZY).

| Fetch Type    | Behavior                                                            |           Example                                                               |
|FetchType.EAGER| Loads the related entity immediately with the parent                | When you fetch a `Patient`, Hibernate also fetches its `Insurance` instantly    |
|FetchType.LAZY|Loads related entity only when accessed (uses proxy object until then)| Fetching Patient, insurance is not loaded until you call `patient.getInsurance()|


‚úÖ Default Fetch Types
| Relationship Type | Default Fetch Type |
| ----------------- | ------------------ |
| @OneToOne         | EAGER              |
| @ManyToOne        | EAGER              |
| @OneToMany        | LAZY               |
| @ManyToMany       | LAZY               |

‚úÖ Example 
If you write:
@OneToOne(fetch = FetchType.LAZY, cascade = CascadeType.ALL)
private Insurance insurance;

‚û§ When you load a patient from DB ‚Üí insurance won‚Äôt be loaded until accessed.
‚û§ When you save a patient ‚Üí insurance will also be saved automatically.

‚û§ Pagination is the process of dividing a large set of data into smaller chunks (pages) to improve performance and user experience.

üß† Why HQL/JPQL Can‚Äôt Perform INSERT Operations
‚û§ HQL (Hibernate Query Language) or JPQL (Java Persistence Query Language) is object-oriented, not table-oriented.
‚û§ It works with entities, not raw database tables.
‚û§ Therefore, HQL doesn‚Äôt support INSERT INTO ... VALUES (...) syntax like SQL does.
‚û§ Instead, it only allows ‚ÄúINSERT INTO ... SELECT ...‚Äù ‚Äî i.e., inserting data by copying from another entity.

‚úÖ Example (the only valid HQL insert form):
@Modifying
@Query("INSERT INTO Patient(name, email, gender, bloodGroup) SELECT p.name, p.email, p.gender, p.bloodGroup FROM OldPatient p")
void copyPatients();

So your original example ‚ùå
@Query("INSERT INTO Patient (name, email, gender, bloodGroup) VALUES (:name, :email, :gender, :bloodGroup)")
üëâ fails because HQL doesn‚Äôt know how to handle literal VALUES insertions ‚Äî that‚Äôs SQL-only.

‚úÖ What You Should Use Instead
1Ô∏è‚É£ ‚Äî Use the Default JPA Repository Method
Simplest and most idiomatic:
Patient patient = new Patient(name, email, gender, bloodGroup);
patientRepository.save(patient);

üëâ Works perfectly with HQL, transactions, and entity lifecycle.
Automatically handles persist/flush operations.

2Ô∏è‚É£ ‚Äî Use Native SQL Query
If you really want custom insertion, switch to native SQL:

@Modifying
@Transactional
@Query(
  value = "INSERT INTO patient (name, email, gender, blood_group) VALUES (:name, :email, :gender, :bloodGroup)",
  nativeQuery = true
)
void insertPatient(@Param("name") String name,
                   @Param("email") String email,
                   @Param("gender") String gender,
                   @Param("bloodGroup") String bloodGroup);
üëâ Works fine because native queries are executed directly as SQL, bypassing HQL limitations.


üß† Soft Delete üÜö Hard Delete
üîÑ Hard Delete
‚û§ The record is permanently removed from the database.
‚û§ Once deleted, the data cannot be recovered.
‚û§ Performed using the standard SQL DELETE statement.

‚úÖ Example:
userRepository.deleteById(1L);

or in SQL:
DELETE FROM users WHERE id = 1;

‚ö° Effect:
The row is completely removed from the users table.

‚ö†Ô∏è Disadvantages:
Lost data cannot be recovered.
No way to maintain historical records or audit trails.

üîÑ Soft Delete
‚û§ The record is not physically deleted from the database.
‚û§ Instead, it is marked as deleted using a flag/column (e.g., is_deleted = true or deleted_at timestamp).
‚û§ The record is ignored in normal queries, but still exists for recovery or auditing.

‚úÖ Example:
üîó User.java Entity Class
@Entity
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private String email;

    private boolean isDeleted = false; // üëà soft delete flag
}

‚úîÔ∏è Repository method:
@Modifying
@Query("UPDATE User u SET u.isDeleted = true WHERE u.id = :id")
void softDelete(@Param("id") Long id);

‚úîÔ∏è Fetch only active users:
@Query("SELECT u FROM User u WHERE u.isDeleted = false")
List<User> findAllActiveUsers();

‚ö° Effect:
Instead of deleting the row, the field is_deleted is updated to true.

üîÅ Compairison Table:
| Feature                  | Hard Delete                 | Soft Delete                     |
| ------------------------ | --------------------------- | ------------------------------- |
| Data physically removed? | ‚úÖ Yes                       | ‚ùå No                            |
| Can be recovered?        | ‚ùå No                        | ‚úÖ Yes                           |
| Query complexity         | Simple                      | Slightly complex (add filter)   |
| Storage use              | Less                        | More                            |
| Best for                 | Irrelevant / Temporary data | Critical / Audit-sensitive data |


üîÑ Advanced Hibernate Trick (Soft Delete Automation)
‚û§ Hibernate provides a way to handle this automatically using:
    ‚úîÔ∏è @SQLDelete ‚Üí custom delete SQL
    ‚úîÔ∏è @Where ‚Üí filter records automatically

‚úÖ Example:
@Entity
@SQLDelete(sql = "UPDATE user SET is_deleted = true WHERE id = ?")
@Where(clause = "is_deleted = false")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;
    private boolean isDeleted = false;
}


üß† Explanation:
‚û§ When you call userRepository.delete(user), Hibernate runs the UPDATE query instead of DELETE.
‚û§ @Where ensures that all SELECT queries automatically ignore soft-deleted rows.


üî∑ Custom Primary Key Generation 
‚û§ In Hibernate/JPA, you can control how the primary key (ID) is generated for your entity.
‚û§ Normally you use:
üîó @GeneratedValue(strategy = GenerationType.IDENTITY) ‚Üí auto-increment
üîó GenerationType.SEQUENCE ‚Üí uses database sequence
üîó GenerationType.AUTO ‚Üí Hibernate decides automatically

‚û§ But custom generation means:
‚û°Ô∏è You define your own logic or sequence for generating IDs ‚Äî not relying on database auto-increment.

üîÑ Methods for Custom ID Generation in Hibernate

1Ô∏è‚É£ Using @GenericGenerator (Hibernate-specific)
This allows you to define your own generator strategy and customize how IDs are formed.
‚úÖ Example: Custom Prefix + Increment ID

Let‚Äôs say you want IDs like:
EMP_1001, EMP_1002, EMP_1003 ...

üîó Code Snippet of Entity Class:
import jakarta.persistence.*;
import org.hibernate.annotations.GenericGenerator;

@Entity
public class Employee {

    @Id
    @GeneratedValue(generator = "custom-id")
    @GenericGenerator(
        name = "custom-id",
        strategy = "org.hibernate.id.enhanced.SequenceStyleGenerator",
        parameters = {
            @org.hibernate.annotations.Parameter(name = "sequence_name", value = "employee_seq"),
            @org.hibernate.annotations.Parameter(name = "initial_value", value = "1000"),
            @org.hibernate.annotations.Parameter(name = "increment_size", value = "1")
        }
    )
    private Long id;
    private String name;
}
üëâ This example uses Hibernate‚Äôs built-in SequenceStyleGenerator but customizes:
    ‚úîÔ∏è Sequence name: employee_seq
    ‚úîÔ∏è Starting value: 1000
    ‚úîÔ∏è Increment: 1
‚û§ This will generate IDs like 1000, 1001, 1002, etc.

üîÑ Explanation:
    ‚úîÔ∏è The parameter sequence_name = "employee_seq" just names the underlying sequence/table, not the prefix of the ID.
initial_value = "1000" means your IDs start at 1000.
    ‚úîÔ∏è increment_size = "1" means each new record increases by 1.
    ‚úîÔ∏è Result = 1000, 1001, 1002, ‚Ä¶, not employee_seq_1000.


2Ô∏è‚É£ Writing a Custom ID Generator Class
If you want full control (e.g., prefix + date + counter), you can create your own generator logic by implementing Hibernate‚Äôs IdentifierGenerator interface.

‚úÖ Example:
‚ö° Step 1: Create Custom Generator
import java.io.Serializable;
import java.util.UUID;
import org.hibernate.engine.spi.SharedSessionContractImplementor;
import org.hibernate.id.IdentifierGenerator;

public class CustomIdGenerator implements IdentifierGenerator {

    @Override
    public Serializable generate(SharedSessionContractImplementor session, Object obj) {
        // Example: custom ID like EMP-2025-<random>
        return "EMP-" + System.currentTimeMillis() + "-" + UUID.randomUUID().toString().substring(0, 4);
    }
}

‚ö° Step 2: Use it in Entity
import jakarta.persistence.*;
import org.hibernate.annotations.GenericGenerator;

@Entity
public class Employee {

    @Id
    @GeneratedValue(generator = "emp-id-gen")
    @GenericGenerator(name = "emp-id-gen", strategy = "com.example.generator.CustomIdGenerator")
    private String id;

    private String name;
}
üëâ Result:
EMP-1739778635191-a7b3
EMP-1739778636234-7c2f
 
üîÑ Hibernate Primary Key Generation Strategies and MySQL Compatibility
| Strategy                       | Description                                                  | Supported in MySQL?   | Notes                                        |
| IDENTITY                       | Uses MySQL‚Äôs auto-increment column                           | ‚úÖ Yes| Most common for MySQL. Each insert gets a new auto ID from DB.|
| SEQUENCE                       | Uses a database sequence object (like in Oracle/PostgreSQL)  | ‚ùå No | MySQL not support sequences natively                          |
| TABLE                          | Uses a separate table to store and increment IDs             | ‚úÖ Yes| Works on any DB, but slower than IDENTITY.                    |
| AUTO                           | Lets Hibernate choose based on dialect.                      | ‚úÖ Yes| In MySQL, it automatically picks `IDENTITY` under the hood.   |
| CUSTOM / @GenericGenerator     | Your own logic or UUID generator.                            | ‚úÖ Yes| Works fine ‚Äî Hibernate handles it internally.                 |

üåü Note: We cannot use generator to generate value for composite primary key.

üîÑ Working with composite Primary Key
To Generate composite primary key we use @Embeddable and @Embedded annotation.
@Embeddable annotation is used for creating a separate table for composite primary key.

üîó AccountPK.java
@Embeddable
public class AccountPK implements Serializable{
    private String accType;
    private Integer accId;

    getters and setters...
}

‚ùì Why Serializable?
1Ô∏è‚É£ For Uniquely Identifying Entity Instances
‚û§ Hibernate needs to store and compare primary key objects during:
    ‚úîÔ∏è Entity caching
    Session management
    ‚úîÔ∏è Lazy loading
    ‚úîÔ∏è Entity identity comparison

‚û§ If your key class is serializable, Hibernate can easily:
    ‚úîÔ∏è Copy it,
    ‚úîÔ∏è Send it over the network,
