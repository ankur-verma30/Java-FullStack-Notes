##############################
🔷 Introduction to PostgreSQL
##############################
PostgreSQL is an open-source, advanced relational database management system (RDBMS) that focuses on:
    ✔️Reliability
    ✔️Data integrity
    ✔️Extensibility
    ✔️Standards compliance

It supports both:
    ✔️ Relational data (tables, rows, columns)
    ✔️ Non-relational data (JSON, arrays, custom types)
👉 Because of this, PostgreSQL is often called an Object-Relational Database Management System (ORDBMS).



❓ Why PostgreSQL is Popular for Backend Systems
PostgreSQL is heavily used in production backend architectures because of its advanced capabilities.

🔄 Major Advantages
1️⃣ Strong ACID Compliance
PostgreSQL guarantees ACID properties:
    ✔️Atomicity
    ✔️Consistency
    ✔️Isolation
    ✔️Durability

This makes it reliable for:
    ✔️Financial systems
    ✔️Banking applications
    ✔️Payment processing
    ✔️Enterprise systems


2️⃣ Advanced Concurrency Control (MVCC)
PostgreSQL uses Multi Version Concurrency Control (MVCC).
This allows:
    ✔️Multiple users to read and write data simultaneously
    ✔️Readers to not block writers
    ✔️Writers to not block readers

✅ Example production scenario:
User A reading order data
User B updating order status
👉 Both operations happen without locking the entire table.


3️⃣ Extensibility
PostgreSQL is highly extensible.
You can add:
    ✔️Custom data types
    ✔️Custom functions
    ✔️Custom operators
    ✔️Custom index types

✅ Example extensions:
    ✔️PostGIS → geospatial queries
    ✔️pg_trgm → fuzzy text search
    ✔️uuid-ossp → UUID generation


4️⃣ Rich Data Type Support
PostgreSQL supports more data types than most databases.
Common types include:
| Type        | Example              |
| ----------- | -------------------- |
| Numeric     | INT, BIGINT, DECIMAL |
| Text        | VARCHAR, TEXT        |
| Boolean     | TRUE / FALSE         |
| Date/Time   | TIMESTAMP            |
| JSON        | JSON, JSONB          |
| Arrays      | INTEGER[]            |
| UUID        | unique identifiers   |
| Range types | INT4RANGE            |


✅ Example:
CREATE TABLE users (
    id UUID PRIMARY KEY,
    name TEXT,
    tags TEXT[],
    metadata JSONB
);

1️⃣ Client
Clients interact with PostgreSQL using:
    ✔️psql (command-line client)
    ✔️Backend applications
    ✔️Database drivers

✅ Examples:
    ✔️Java → JDBC
    ✔️Node.js → pg
    ✔️Python → psycopg
    ✔️Go → pgx
    
🔗 Example connection: psql -U postgres -d mydatabase


2️⃣ Query Parser
The parser:
    ✔️Checks SQL syntax
    ✔️Validates table and column names
    ✔️Converts SQL into parse tree

✅ Example query: SELECT * FROM users WHERE id = 10;
👉 PostgreSQL converts it internally into a structured representation.


🔄 Query Planner / Optimizer
This is one of PostgreSQL's most powerful components.
It decides:
    ✔️Which index to use
    ✔️Whether to perform a sequential scan
    ✔️Which join algorithm to use

Possible execution strategies:
    ✔️Sequential Scan
    ✔️Index Scan
    ✔️Bitmap Index Scan
    ✔️Hash Join
    ✔️Merge Join
    ✔️Nested Loop Join

✅ Example: EXPLAIN SELECT * FROM users WHERE id = 10;
👉 The planner chooses the most efficient query execution plan.


🔄 Query Executor
The executor:
    ✔️Executes the plan generated by the planner
    ✔️Retrieves rows from storage
    ✔️Returns results to the client


🔄 Storage Engine
The storage layer manages:
    ✔️Tables
    ✔️Indexes
    ✔️Transactions
    ✔️Write Ahead Logs (WAL)

Important characteristics:
    ✔️Heap storage model
    ✔️MVCC-based concurrency
    ✔️Crash recovery through WAL

🔄 PostgreSQL vs Other Databases
PostgreSQL vs MySQL
| Feature              | PostgreSQL            | MySQL                |
| -------------------- | --------------------- | -------------------- |
| Standards compliance | Very high             | Moderate             |
| JSON support         | JSONB (very powerful) | JSON (less powerful) |
| Extensibility        | Very high             | Limited              |
| Concurrency model    | MVCC                  | MVCC (InnoDB only)   |
| Query optimizer      | More advanced         | Simpler              |


PostgreSQL can often replace NoSQL databases because of:
    ✔️JSONB
    ✔️Flexible schema
    ✔️Indexing JSON fields

✅ Example:

FROM users
WHERE metadata->>'country' = 'India';

##################################
🔷 CRUD Operations in PostgreSQL
##################################
CRUD stands for:
✔️ C → Create
✔️ R → Read
✔️ U → Update
✔️ D → Delete
👉 In PostgreSQL, these operations are performed using SQL commands:

| Operation | SQL Command |
| --------- | ----------- |
| Create    | `INSERT`    |
| Read      | `SELECT`    |
| Update    | `UPDATE`    |
| Delete    | `DELETE`    |


🔄 CREATE Operation (INSERT)
The Create operation inserts new records into a table.

🔗 SQL Command
INSERT INTO table_name (column1, column2, column3)
VALUES (value1, value2, value3);

✅ Example Table
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    email TEXT UNIQUE,
    age INT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

➤ Basic Insert
INSERT INTO users (name, email, age)
VALUES ('Ankur', 'ankur@example.com', 25);

➤ Insert Multiple Rows
INSERT INTO users (name, email, age)
VALUES
('Rahul', 'rahul@example.com', 27),
('Aman', 'aman@example.com', 30);
Insert Returning Data (Very Common in Backend APIs)

➤ PostgreSQL supports RETURNING, which is extremely useful.
✅ Example:
INSERT INTO users (name, email, age)
VALUES ('Rohit', 'rohit@example.com', 29)
RETURNING id;

Backend usage:
✔️ Create user
✔️ Immediately return generated ID

1️⃣ Constraints
During insertion PostgreSQL checks constraints.
✅ Examples:
    ✔️ PRIMARY KEY
    ✔️ UNIQUE
    ✔️ NOT NULL
    ✔️ CHECK
    ✔️ FOREIGN KEY

✅ Example failure:
INSERT INTO users (name)
VALUES (NULL);
👉 Fails because of NOT NULL constraint.


2️⃣ Write Ahead Logging (WAL)
Every insert is written to WAL first before disk.

Purpose:
    ✔️ Crash recovery
    ✔️ Data durability

Flow:
Insert Query
   ↓
Write to WAL
   ↓
Data written to table


3️⃣ MVCC Version Creation
When a row is inserted:
    ✔️ PostgreSQL creates a tuple version
    ✔️ This version is visible depending on transaction isolation


4️⃣ Default Values
PostgreSQL automatically fills default values.

✅ Example:
created_at = CURRENT_TIMESTAMP


🔄 READ Operation (SELECT)
➤ The Read operation retrieves data from tables.
➤ This is the most frequent operation in backend systems.

🔁 Basic Select
🔗 SELECT * FROM users;

🔁 Select Specific Columns
🔗 SELECT name, email
FROM users;

🔁 Filtering Data
🔗 SELECT *
FROM users
WHERE age > 25;

🔁 Sorting Results
🔗 SELECT *
FROM users
ORDER BY age DESC;

🔁 Limiting Results (Pagination)
➤ Very common in APIs.
🔗 SELECT *
FROM users
LIMIT 10 OFFSET 20;


1️⃣ Query Planner
➤ PostgreSQL decides the best execution plan.
➤ Possible strategies:
    ✔️ Sequential Scan
    ✔️ Index Scan
    ✔️ Bitmap Index Scan
✅ Example: EXPLAIN SELECT * FROM users WHERE id = 5;


2️⃣ Index Usage
If an index exists: CREATE INDEX idx_users_email ON users(email);
🔗 Query: SELECT * FROM users WHERE email = 'ankur@example.com';
👉 PostgreSQL uses Index Scan.


3️⃣ MVCC Visibility
➤ When reading data PostgreSQL checks: transaction ID visibility
➤ This ensures:
    ✔️ readers do not block writers
    ✔️ writers do not block readers

✅ Example scenario:
Transaction A updating a row
Transaction B reading table
👉 Transaction B still reads previous version of row.


4️⃣ Caching (Shared Buffers)
➤ PostgreSQL stores frequently accessed data in: shared_buffers
➤ This improves performance for repeated reads.


🔄 UPDATE Operation
Update modifies existing rows.

🔁 Basic Update
🔗 UPDATE users
SET age = 26
WHERE id = 1;

🔁 Update Multiple Columns
🔗 UPDATE users
SET name = 'Ankur Verma',
    age = 26
WHERE id = 1;

🔁 Update with Condition
🔗 UPDATE users
SET age = age + 1
WHERE age < 30;

🔁 Update Returning Updated Row
🔗 UPDATE users
SET age = 28
WHERE id = 1
RETURNING *;

1️⃣ MVCC
PostgreSQL does not overwrite rows.
Instead:
Old Row Version → marked as dead
New Row Version → inserted

✅ Example:
Row version 1 → age = 25
Row version 2 → age = 26
👉 Old row remains until VACUUM cleans it.


2️⃣ Row Level Locking
When updating:
    ✔️ Row-level lock is acquired
    ✔️ Other transactions cannot update the same row.

✅ Example conflict:
Transaction A updating user id=1
Transaction B updating same row
Transaction B must wait.

3️⃣ Index Updates
If indexed column is updated:
    ✔️ PostgreSQL updates index entries
    ✔️ This can affect performance.

4️⃣ Write Ahead Logging
Updates are recorded in WAL for durability.


🔄 DELETE Operation
Delete removes rows from a table.

🔁 Basic Delete
🔗 DELETE FROM users
WHERE id = 5;

🔁 Delete Multiple Rows
🔗 DELETE FROM users
WHERE age < 18;

🔁 Delete All Rows
🔗 DELETE FROM users;

🔁 Better option:
🔗 TRUNCATE TABLE users;

🔁 Delete Returning Deleted Rows
🔗 DELETE FROM users
WHERE id = 5
RETURNING *;


1️⃣ MVCC Soft Delete
➤ PostgreSQL does not immediately remove rows.
➤ Instead:
    ✔️ Row is marked as deleted
    ✔️ The row becomes invisible to new transactions.
➤ Actual cleanup happens via: VACUUM


2️⃣ Row Level Lock
➤ Deleting a row locks it.
➤ Other transactions: cannot update or delete same row


3️⃣ Foreign Key Constraints
Delete may fail if foreign key exists.
✅ Example: orders.user_id → users.id
👉 Deleting user might fail.


4️⃣ Cascading Deletes
✅ Example:
FOREIGN KEY (user_id)
REFERENCES users(id)
ON DELETE CASCADE
👉 Deleting user automatically deletes related rows.
